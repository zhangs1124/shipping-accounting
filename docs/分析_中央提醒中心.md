# 中央提醒中心 (Central Reminder Center) 需求分析與設計提案

## 1. 核心需求分析

根據您的要求，我們將「中央提醒中心」定位為一個跨功能的通知導航站，首要任務是處理「進出港檢查項目」的時效性提醒。

### 1.1 使用者維度 (Personas)
- **輸入人員 (Operators)**：負責執行進出港任務。他們應在登入後看到「屬於自己負責」的逾期或即將到期提醒。
- **管理人員 (Managers)**：負責監控全體進度。他們應能看到「所有人員」的提醒狀態，並能依人員進行篩選。

### 1.2 提醒觸發機制：進出港檢查項目
目前 `VoyageTaskLog` 僅記錄「何時完成」，缺乏「應何時完成」的基準。
- **提案**：在 `TaskCategory` (任務類別) 增加 `expected_offset_hours` (預計偏移小時) 與 `base_milestone` (基準點：如 ETA 或 ETD)。
- **範例**：
  - 「預報進港 (Arrival Notice)」：基準點為 ETA，偏移 -24 小時。
  - 「申請引水 (Pilotage)」：基準點為 ETA，偏移 0 小時。
  - 「檢疫完成 (Quarantine)」：基準點為 ETA，偏移 +2 小時。

## 2. 資料庫設計提案

### 2.1 新增 `Reminders` 資料表
用來存放所有產生的提醒訊息，供介面顯示。

| 欄位 | 型別 | 說明 |
| :--- | :--- | :--- |
| `id` | Integer (PK) | |
| `title` | String | 提醒標題 (例：引水時間逾期) |
| `content` | Text | 提醒詳細描述 |
| `remind_type` | String | 類別 (TASK_OVERDUE, TASK_UPCOMING, SYSTEM) |
| `source_table` | String | 來源表名 (voyage_task_logs) |
| `source_id` | Integer | 來源 ID |
| `target_employee_id` | Integer (FK) | **(關鍵)** 指向此項目的負責人 |
| `is_closed` | Boolean | 是否已處理結案 |
| `deadline` | DateTime | 預計應完成的時間 |
| `created_at` | DateTime | 產生提醒的時間 |

### 2.2 調整 `VoyageTaskLog` 或 `Voyage`
- 目前 `VoyageTaskLog` 沒有 `assigned_to`，建議在初始化任務時，或在 `Voyage` 主檔中明確定義「負責外勤人員」。

---

## 3. 功能介面規劃

### 3.1 首頁/提醒看板 (`/reminders`)
- **普通人員**：僅列出 `target_employee_id = 自己` 且 `is_closed = False` 的項目。
- **管理人員**：預設列出全部項目，並提供「依人員」下拉選單過濾。

### 3.2 自動產生邏輯 (小秘書任務)
- 每 15~30 分鐘執行一次掃描。
- 比對 `Voyage.ETA/ETD` + `TaskCategory.offset` 是否小於「現在」。
- 若 `recorded_time` 仍為空，且 `Reminders` 中尚未存在該任務的未結案提醒，則新增一筆。
