# 分析：操作紀錄 (Audit Log)

## 1. 討論背景與目標
在確保資料庫可以循環備份的前提下，為了能針對單筆資料的誤刪除、誤修改進行追蹤與還原，系統需要導入「操作紀錄 (Audit Log)」。這也是財務會計系統中標準的防呆與稽核機制。

## 2. 紀錄範圍與觸發時機

### 2.1 基礎資料 CRUD (全自動攔截)
系統中的核心資料表（如同 `Voyage`, `Invoice`, `Employee` 等），原則上都需要紀錄以下動作：
- **新增 (CREATE)**：紀錄新增的完整資料內容。
- **修改 (UPDATE)**：紀錄修改前的舊資料 (`old_value`) 以及修改後的新資料 (`new_value`)，以利比對差異。
- **刪除 (DELETE)**：紀錄被刪除的完整舊資料 (`old_value`)，作為未來反悔還原的依據。

**實作考量**：
此部分將採用 SQLAlchemy 的 Event Listeners (`after_insert`, `after_update`, `after_delete`) 來實作，盡可能作到對應用層透明（即開發業務邏輯時不需要手動寫程式紀錄）。

### 2.2 特殊業務行為 (手動寫入)
除了一般的資料庫 CRUD，某些重要的業務行為並非單純的單表異動，也可能需要留下足跡：
- **登入 (LOGIN) / 登出 (LOGOUT)**：紀錄誰在何時、哪個 IP 進行了系統登入。
- **寄發通知 (SEND_EMAIL)**：例如系統自動或手動寄發逾期帳單通知。
  - *分析*：寄發通知雖然可能只是標記 `is_reminded = True`，但「寄信」本身也是一項重要通訊行為。如果系統後續發生爭議（客戶說沒收到信），Audit Log 可以作為系統確實有觸發寄信排程的證明。
  - *建議*：寄發通知**應該**寫入 Audit Log，動作名稱可訂為 `SEND_EMAIL`。

## 3. 欄位設計要求
- `user_id`: 觸發該操作的使用者。若是系統排程（如自動寄信），則標記為系統。
- `action`: 此筆紀錄的類型（`CREATE`, `UPDATE`, `DELETE`, `LOGIN`, `SEND_EMAIL` 等）。
- `table_name`: 關聯的資料表名稱。
- `target_id`: 關聯的資料主鍵 ID。
- `old_value` / `new_value`: JSON 格式，儲存操作前後的詳細資料。
- `timestamp`: 操作發生的時間。
- `ip_address`: 來源 IP。

## 4. 階段實作策略
1. **第一階段 (完善基礎功能)**：
   - 先建立 `AuditLog` 資料表與 Python Model。
   - 建立共用的寫入輔助函數。
   - 實作 SQLAlchemy 事件攔截器，捕捉所有資料庫表的新增、刪除、修改。
2. **第二階段 (討論與精細化)**：
   - 確認哪些特定的功能 API 需要手動埋點（例如登入登出、發送 Email 等）。
   - 解決背景排程（無 HTTP Request）時，`user_id` 與 `ip_address` 如何定義的問題。
