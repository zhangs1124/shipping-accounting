# 分析：系統操作紀錄時間差異 (時區問題)

## 1. 問題現象
您在查看「系統稽核與操作紀錄 (Audit Log)」時，發現上面記錄的時間與您電腦的實際時間（台北林區時間，也就是 UTC+8）有落差。通常時間會比現在的時間**早了 8 小時**。

## 2. 根本原因 (Root Cause)

經過我的調查，這個問題是因為 Python (`datetime.now()`) 與資料庫 (`SQLite`) 計算預設時間的方式不同所造成的：

### A. SQLite 的 `func.now()`
在我們的資料庫設計檔 `models.py` 中，很多時間欄位（包含 `AuditLog.timestamp`、所有表的 `created_at` 等）都使用以下寫法設定預設值：
```python
timestamp = Column(DateTime, default=func.now())
```
`func.now()` 會直接呼叫資料庫底層的函數。而 **SQLite 的預設 `CURRENT_TIMESTAMP` 是嚴格採用 UTC（世界協調時間）**的。當它把時間存入資料庫時，它存的是 UTC 時間，而非您伺服器的台北時間。

### B. `audit_logger.py` 的實作疏漏
我們在 `audit_logger.py` 中自動監聽資料表的異動（CREATE, UPDATE, DELETE），並生成 AuditLog 紀錄。但在產生這些紀錄時：
```python
audit_entries.append(AuditLog(
    user_id=uid,
    action="UPDATE",
    # 注意：這裡我們沒有給 timestamp 參數
))
```
因為我們沒有給 `timestamp` 參數，SQLAlchemy 自動調用了 `func.now()` 交給 SQLite 去生成時間，導致該筆紀錄被標記為 UTC 時間。

**(相對的，我們在發送信件 `SEND_EMAIL` 的手動埋點時，有些時候有傳入 Python 的 `datetime.now()`，所以那些紀錄可能會是正確的台北時間，造成系統內時間標準不一致)**。

## 3. 建議解決方案

為了保持系統時間的一致性，且確保使用者在介面上看到的是習慣的本地時間（台北時間 UTC+8），這有兩種常見解法：

### 方案一：全面改用 Python 伺服器時間 (最簡單、最直觀)
將 `models.py` 中所有的 `func.now()` 改為使用 Python 的 `datetime.now`。
改法就像這樣：
```python
from datetime import datetime

# 修改前
created_at = Column(DateTime, default=func.now())

# 修改後
created_at = Column(DateTime, default=datetime.now)
```
**優點**：Python 的 `datetime.now()` 會抓取作業系統的時區（也就是您的台北時間），並且在塞入資料庫前就已經是正確的時間字串。前端介面也不用做任何特殊的轉換，直接顯示即可。

### 方案二：全面套用 UTC，統一由前端處理
將系統內部全部時間嚴格遵守 UTC 儲存，但在前端的 `.html` 檔案或 JavaScript 中，使用 `Intl.DateTimeFormat` 或 Moment.js 轉換成使用者瀏覽器所在地的時區。
**缺點**：要改動的前端頁面與邏輯較多，工程稍大。
