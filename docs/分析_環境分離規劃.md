# 開發與正式環境分離規劃分析

## 1. 需求背景與現況
目前系統都是透過執行單一的 `run_app.bat` 來啟動：
```bat
python -m uvicorn main:app --reload --host 0.0.0.0 --port 8000
```
**現況痛點**：
- **綁定開發模式**：使用了 `--reload`（代碼一改動就重啟），這在正式上線 (Production) 環境會導致不穩定，且消耗較多效能。
- **配置固定**：Port 寫死為 `8000`。
- **資料庫共用**：如果開發與上線都在同一個目錄中執行，會同時讀寫同一個 `shipping.db`，開發時如果弄壞資料，會直接影響正式環境。

## 2. 目標：完全隔離開發與線上環境
要達到「開發歸開發、正式歸正式，Port 不同且互相不干擾」的目標，可以採取以下幾個層級的分離：

### Step A: 實體執行個體分離 (資料夾維度)
最安全且簡單的做法是在伺服器上建立兩個不同的資料夾：
- **開發區 (Dev)**：`D:\project\Voyage\shipping-accounting-dev`
  - 供 Antigravity 與開發者日常寫 Code、測試使用。
- **正式區 (Prod)**：`D:\project\Voyage\shipping-accounting-prod`
  - 只有在開發區測試完美後，才透過 Git (如 `git pull`) 或是複製覆蓋的方式更新這個資料夾的代碼。

### Step B: 啟動腳本與 Port 分離
針對開發與正式環境，建立不同的啟動檔案：

1. **`run_dev.bat` (開發環境)**
   ```bat
   @echo off
   echo Starting DEV Environment...
   python -m uvicorn main:app --reload --host 0.0.0.0 --port 8000
   ```
   *(Port 8000, 啟用 reload 方便修改檢視)*

2. **`run_prod.bat` (正式環境)**
   ```bat
   @echo off
   echo Starting PROD Environment...
   python -m uvicorn main:app --host 0.0.0.0 --port 8080
   ```
   *(Port 8080, 關閉 reload 以確保穩定，並可考慮加入 Gunicorn 或 Uvicorn workers 來提升效能)*

### Step C: 資料庫與環境變數分離 (`.env`)
如果堅持要在**同一個資料夾**啟動兩個不同的系統（不建議，極不安全），或是為了更靈活地控制，我們需要引入 `.env` 機制：
- **Dev 環境 `.env`**：
  ```env
  APP_ENV=development
  DB_URL=sqlite:///./shipping_dev.db
  ```
- **Prod 環境 `.env`**：
  ```env
  APP_ENV=production
  DB_URL=sqlite:///./shipping.db
  ```

這樣程式中就可以透過讀取 `APP_ENV` 來決定連接哪顆資料庫、隱藏/顯示詳細報錯等。

## 3. 發佈流程建議 (Workflow)
未來您的開發到上線流程應該是：
1. **開發階段**：啟動 `run_dev.bat`（Port 8000），使用開發版資料庫，不斷對系統進行修改與測試。
2. **版本控制**：確認測試沒問題後，使用 `git commit` 將異動存擋。
3. **佈署正式版**：
   - 到正式版資料夾（或透過工具）拉取最新代碼。
   - 正式版本身正默默跑在 Port 8080，重啟 `run_prod.bat` 讓新代碼生效。
