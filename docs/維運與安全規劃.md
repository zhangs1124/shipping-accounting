# 航運會計系統：資料安全與稽核規劃 (Data Security & Audit Plan)

本文件詳述系統針對 SQLite 資料庫的併發處理、循環備份以及操作稽核的規劃。

## 1. 資料庫併發優化 (SQLite WAL)
- **模式**：已切換至 `Write-Ahead Logging (WAL)` 模式。
- **優勢**：
  - 讀寫分離：讀取操作不會被寫入操作阻塞。
  - 提升效能：在高頻率讀寫下減少 `database is locked` 錯誤。
  - 增強穩定性：具備更佳的當機自動復原能力。

## 2. 循環備份規劃 (Rotation Backup)
為確保資料具備「時光回溯」能力，規劃如下：
- **備份頻率**：每 1 小時執行一次完整備份。
- **實作方式**：
  - 使用 SQLite 內建 `backup` API 進行線上備份（確保包含 WAL 中未合併的資料）。
  - 備份檔名格式：`shipping_backup_YYYYMMDD_HH.db`。
- **保留策略 (Retention)**：
  - 保留最近 **10 天** 的備份檔案。
  - 超過 10 天的舊檔案將由自動化腳本自動刪除。
- **執行機制**：整合至現有 `APScheduler` 定時任務系統。

## 3. 操作紀錄規劃 (Audit Log)
為滿足財務稽核與精確還原需求，規劃建立 `AuditLog` 機制。
- **紀錄範圍**：
  - 所有的 `INSERT` (新增)、`UPDATE` (修改)、`DELETE` (刪除) 動作。
  - 登入、登出等安全相關行為。
- **核心欄位**：
  - `user_id`: 執行操作的使用者帳號。
  - `action`: 動作類型 (CREATE/UPDATE/DELETE)。
  - `table_name`: 受影響的資料表（如 `invoices`, `voyages`）。
  - `target_id`: 被異動物件的 ID。
  - `old_value`: 變更前資料 (JSON 格式)，可用於手動回溯。
  - `new_value`: 變更後資料 (JSON 格式)。
  - `timestamp`: 操作精確時間。
  - `ip_address`: 使用者來源 IP。
- **實作方式**：
  - 於 SQLAlchemy Model 層級加入整合。
  - 每次資料異動時同步寫入 `audit_logs` 資料表。
