# 中央提醒中心 (Central Reminder Center) 系統規劃文件

> 建立日期：2026-03-26
> 狀態：**規劃中 (暫緩實作，等候登入系統完成後銜接)**

---

## 一、需求背景

目前系統已有「帳單逾期提醒」的自動 Email 功能，但隨著業務擴大，需要一套更具彈性的提醒架構，能夠：
1. 對「進出港任務關鍵項目」設定個別提醒（例如：「RPM 時間」若 3 天內未填，就發 Email 警告）。
2. 支援自訂的、非系統自動產生的備忘錄提醒（例如：「下週三要打電話給 A 客戶」）。
3. 提供一個統一的提醒管理頁面，讓每位使用者能自行管理自己的提醒任務。
4. **此提醒系統與帳單提醒邏輯完全獨立**，未來方便個別調整。

---

## 二、核心設計原則

- **中央化**：所有來源（進出港、帳單、手動）的提醒，統一存入同一張 `Reminders` 資料表，排程器只需巡邏這一張表。
- **個人化**：每筆提醒會綁定「負責人」(`assigned_to`) 與「建立者」(`created_by`)，讓每位使用者只看到自己的任務。
- **可重複催收**：同一筆提醒可設定「每隔幾天重複提醒」，直到被標記為「已結案」為止。

## 三、資料庫設計

### 3.1 資料表：`Reminders`

| 欄位名稱 | 資料型別 | 說明 |
|---|---|---|
| `id` | Integer (PK) | 流水號 |
| `title` | String | 提醒主旨（例如：「EG-2024-001 引水時間未填寫」）|
| `description` | Text | 詳細說明或備忘內容（可選）|
| `source_type` | String | 來源類別：`VOYAGE_TASK` / `INVOICE` / `CUSTOM` |
| `source_id` | Integer | 來源資料的 ID（例如：進出港任務的 voyage_id），方便生成超連結 |
| `remind_at` | DateTime | 首次提醒的時間 |
| `interval_days` | Integer | 每隔幾天重複提醒（`0` 或 `NULL` 代表單次提醒）|
| `last_reminded_at` | DateTime | 上次實際發送提醒的時間（排程器用來判斷是否已過間隔） |
| `remind_count` | Integer | 已提醒次數（可選，用於限制最大提醒次數）|
| `max_remind_count` | Integer | 最多提醒幾次即放棄（`NULL` 代表無限制）|
| `status` | String | 目前狀態：`處理中` / `已結案` / `已忽略` |
| `assigned_to` | Integer (FK -> Users) | 負責處理此提醒的人員 ID（未來登入系統提供） |
| `created_by` | Integer or String | 建立者（可以是使用者 ID，或固定字串 `System` 代表自動產生）|
| `created_at` | DateTime | 建立時間 |
| `closed_at` | DateTime | 結案時間（結案後填入）|

## 四、排程器邏輯設計

小秘書 (APScheduler) 在背景巡邏的 SOP：

```
1. 查詢所有 status = '處理中' 的提醒任務。

2. 對每一筆進行判斷：
   a. 若 last_reminded_at IS NULL，且 remind_at <= 現在  
      → 【首次提醒】發信！
   
   b. 若 interval_days > 0，且 (現在 - last_reminded_at) >= interval_days 天
      → 【重複催收】發信！
   
   c. 若 max_remind_count 不為空，且 remind_count >= max_remind_count
      → 跳過，不再發信（可選擇自動標記為 '已忽略'）

3. 每次發信後：
   - 更新 last_reminded_at = 現在時間
   - remind_count += 1
   - 不自動變更 status（要等業務別去結案）

4. 發信目標：
   - 若 assigned_to 有綁定使用者，就寄到該使用者的 email
   - 若無，回退寄到 .env 的預設 NOTIFICATION_RECIPIENT
```

## 五、進出港任務的整合方式 (Future Integration)

1. 在「進出港任務詳細頁」(`detail.html`) 為每個任務項目加上一個「🔔 設定提醒」按鈕。
2. 點擊後彈出小視窗，設定：
   - **首次提醒時間**（例如：幾天後）
   - **是否重複催收** 以及 **間隔天數**
3. 後端接收到設定後，在 `Reminders` 表建立一筆新資料，並將 `source_type` 設為 `VOYAGE_TASK`、`source_id` 設為該任務的 `VoyageTaskLog.id`。
4. **結案機制**：當員工在進出港詳細頁填入該任務的「執行時間」時，後端同步查詢 `Reminders` 是否存在對應的待辦，若存在則自動將其 `status` 標記為 `已結案`。

## 六、管理介面規劃

### 6.1 「我的提醒看板」頁面 (Personal Reminder Board)
- **路由**：`/reminders`
- **存取邏輯**：登入後，只顯示 `assigned_to = 目前登入者` 的提醒。
- **功能**：
  - 顯示「處理中」的提醒列表（含來源連結）。
  - 每筆可操作：「標記完成」/ 「忽略」/ 「修改提醒時間」。
  - 新增「自訂提醒」按鈕，供建立 `source_type = CUSTOM` 的個人備忘錄。

### 6.2 管理員總覽 (Admin View) [進階需求]
- 可看到全公司所有提醒的狀態，方便主管追蹤整體進度。
- 可過濾：`source_type`、`assigned_to`、`status`。
