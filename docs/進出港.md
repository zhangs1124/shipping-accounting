這是一個非常典型的 「航次任務檢核清單 (Voyage Milestone/Checklist)」 邏輯。在船務代理實務中，船舶進出港有一連串標準作業程序（SOP），每一項動作的時間點對於後續的規費計算、延滯費判斷以及申報合規性至關重要。

既然您目前只需要「紀錄時間」而不需要實體檔案，我們應該將這些動作抽象化為 「動態任務」。這樣未來如果主管想增加一個「領港上下船時間」，您不需要改程式碼，只需要在基本檔加一筆資料即可。

一、 業務邏輯分析
根據您提供的清單，我將其歸納為三個主要維度：

任務類別維度 (Task Categories)：

申報類： 進/出港預報、重報、SDH 上傳。

檢疫類： 船舶檢疫證明書、SSCEC、買/換准單。

物流/現場類： 引水時間、RPM。

財務/單據類： 列印繳費單 ($1058)、下載三連單。

時序性與重複性：

進出港對稱： 進港有預報，出港也有預報。

重報機制： 預報時間可能會變動，所以會有「重報時間」，這代表同一個任務可能會有多次紀錄。

核心需求：

精確紀錄： 紀錄動作發生的精確秒數。

責任追蹤： 誰在什麼時候點擊了這個動作？

二、 SQL 資料庫設計
為了滿足「類別基本檔」與「時間紀錄」的需求，建議增加以下兩張表：

SQL
-----------------------------------------------------------
-- 1. 任務/文件動作類別基本檔 (Master Data)
-----------------------------------------------------------
CREATE TABLE TaskCategories (
    TaskID INT PRIMARY KEY IDENTITY(1,1),
    TaskName NVARCHAR(100) NOT NULL,    -- 任務名稱 (如：進港預報、買檢疫准單)
    TaskGroup NVARCHAR(50),             -- 分組 (如：檢疫、預報、財務)
    DefaultFee DECIMAL(18, 0) DEFAULT 0, -- 預設規費 (如超商繳費 $1058)
    DisplayOrder INT DEFAULT 0,         -- 顯示順序
    IsActive BIT DEFAULT 1
);

-----------------------------------------------------------
-- 2. 航次任務執行時間紀錄表 (Transaction Data)
-----------------------------------------------------------
CREATE TABLE VoyageTaskLogs (
    LogID INT PRIMARY KEY IDENTITY(1,1),
    VoyageID INT NOT NULL,              -- 關聯航次
    TaskID INT NOT NULL,                -- 關聯任務類別
    RecordedTime DATETIME NOT NULL DEFAULT GETDATE(), -- 實際動作時間
    RecordedBy INT NOT NULL,            -- 執行的使用者
    Remarks NVARCHAR(500),              -- 備註 (例如：重報原因)
    
    FOREIGN KEY (VoyageID) REFERENCES Voyages(VoyageID),
    FOREIGN KEY (TaskID) REFERENCES TaskCategories(TaskID),
    FOREIGN KEY (RecordedBy) REFERENCES Users(UserID)
);
三、 功能實作方向建議
1. 前端 UI (Bootstrap)
在您的航次編輯頁面中，可以設計一個「任務面板」：

左側顯示任務名稱。

中間顯示「已紀錄時間」（若未紀錄則顯示「尚未執行」）。

右側放一個 [紀錄目前時間] 的按鈕。

如果是「列印繳費單」，點擊時除了紀錄時間，還可以順便跳出視窗讓使用者確認金額是否為 $1058。

2. C# 後端邏輯
當使用者點擊按鈕時，透過 Ajax 呼叫 Controller：

C#
[HttpPost]
public JsonResult RecordTaskTime(int voyageId, int taskId, string remark)
{
    // 1. 檢查航次是否已鎖定
    if (IsVoyageLocked(voyageId)) 
        return Json(new { success = false, message = "航次已結案，無法新增動作紀錄。" });

    // 2. 寫入 VoyageTaskLogs
    var log = new VoyageTaskLog {
        VoyageID = voyageId,
        TaskID = taskId,
        RecordedTime = DateTime.Now,
        RecordedBy = CurrentUser.UserID,
        Remarks = remark
    };
    db.VoyageTaskLogs.Add(log);
    db.SaveChanges();

    return Json(new { success = true, time = log.RecordedTime.ToString("yyyy-MM-dd HH:mm:ss") });
}
四、 進階分析：為什麼這樣設計更好？
擴充性高： 航港局如果改了法規，多了一個「壓艙水申報時間」，您只要在 TaskCategories 加一筆，網頁就會自動長出那個按鈕。

支援重報： 透過 VoyageTaskLogs 的一對多設計，您可以清楚看到進港預報「第一次是什麼時候」、「第二次重報是什麼時候」，這對於釐清作業延遲責任非常有幫助。

規費聯動： 您提到的「超商繳費 $1058」，因為我們在 TaskCategories 存了 DefaultFee，未來您可以寫一個功能：「自動根據已執行的任務產出記帳細項」。例如：只要有「列印繳費單」的紀錄，系統就自動在帳單明細加一筆 $1058 的規費支出。

這套邏輯能把「作業流程」跟「財務記帳」完美掛鉤。
