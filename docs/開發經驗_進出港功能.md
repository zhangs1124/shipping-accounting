# 開發經驗紀錄 - 進出港 SOP 任務檢核功能

## 1. 需求背景
船務代理作業中，船舶進出港有一套標準作業程序 (SOP)。精確紀錄這些動作的時間點對於後續的規費計算與責任追蹤至關重要。

## 2. 技術設計
### 2.1 資料庫設計 (SQLAlchemy)
- **TaskCategory**: 儲存任務的主檔，支援分組 (Task Group) 與預設規費 (Default Fee)。
- **VoyageTaskLog**: 交易紀錄表，採用一對多設計，支援多次重報紀錄，但介面僅呈現最新的一筆。

### 2.2 前端互動 (Vanilla JS)
- 採用 **AJAX (Fetch API)** 進行非同步記錄。
- **彈窗式紀錄 (Modal)**: 點擊按鈕後彈出 Modal，預設自動填入系統時間。
- **時間選擇與清除**: 使用者可在 Modal 中手動調整時間，介面更為乾淨且易於確認。
- 使用動畫提示 (Green Flash) 增強使用者反饋。

## 3. 實作重點
- **任務分組**: 根據文件需求，將任務分為「預報」、「現場」、「檢疫」、「財務」、「申報」與「單據」六大類，使介面結構清晰。
- **防止误觸**: 加入了 JavaScript `confirm` 確認視窗。
- **可擴展性**: 若未來需要增加新任務，只需在 `task_categories` 資料表中新增即可，前端介面會自動動態渲染。

## 4. 後續規劃
- **帳務自動關聯**: 未來可根據已記錄的任務（例如：列印繳費單），自動在該航次的帳單明細中產生對應的規費項目。
- **文件上傳**: 目前僅紀錄時間，未來可擴充實體文件的上傳與預覽功能。

# 第二次改版紀錄 (2026-03-25)

## 需求變更
使用者希望介面改為與費用帳單相同的操作流程：
1. 上方輸入**船名 + 航次編號**後按「新增」建立一筆進出港記錄
2. 系統自動帶入所有啟用的檢查項目
3. 進入詳細頁可以動態**加入**或**移除**檢查項目（和費用明細一樣）

## 設計決策

### 拆成列表頁 + 詳細頁
- 原本只有一個頁面（`list.html`），改版後拆為：
  - `list.html`：顯示所有已建進出港記錄的航次 + 上方新增表單
  - `detail.html`（新建）：單一航次的檢查清單，支援加減項目

### 船名輸入使用 `<datalist>` autocomplete
- 使用 HTML5 原生 `<datalist>` 搭配 `<input list="...">` 提供船名下拉建議
- 後端以船名或代碼模糊匹配，容錯性更高

### 新增記錄流程 (`POST /create`)
- 接收 `ship_name` + `voyage_no`
- 若該航次尚無任何 `VoyageTaskLog`，自動初始化所有啟用中的 `TaskCategory`
- 已有記錄的航次再次新增，直接導向詳細頁（不重複初始化）

### 列表頁顯示完成進度條
- 計算 `done / total`，呈現視覺化進度條
- 顏色：未完成(灰)、部分完成(黃)、全完成(綠)

### 動態加減（參考費用模組）
- **移除**：`POST /api/delete/{log_id}` → 刪除後 `location.reload()` 更新下拉選單
- **新增**：`POST /api/add` → 新增後 `location.reload()` 更新已用/可用清單

## 修改的檔案
| 檔案 | 異動 |
|------|------|
| `routers/voyage_tasks.py` | 全面改版：新增 list/create/detail 路由 |
| `templates/voyage_tasks/list.html` | 改版：輸入表單＋進度清單 |
| `templates/voyage_tasks/detail.html` | **新增**：任務表格＋加減功能＋紀錄時間 Modal |

### 航次連動選單優化 (2026-03-25)
- **需求**：新增進出港記錄時，航次應從下拉選單選擇。
- **實作**：
  - 新增後端 API `/voyage-tasks/api/voyages` 依船名查詢航次。
  - 前端 `list.html` 監控船名輸入，當匹配成功時觸發 Fetch 取得航次並填充至 `<select>` 選單。
  - 提升了資料錄入的準確性，避免手打航次編號造成的錯誤。

### 初始化與清除功能優化 (2026-03-25)
- **需求**：新增記錄後不應自動帶入目前時間，且需要能清除已填寫的時間。
- **實作**：
  - 修改 `models.py`：移除 `VoyageTaskLog.recorded_time` 的 `default=func.now()`，使新任務初始為空值。
  - 修改 `detail.html`：在操作列新增「✕」(清除) 按鈕。
  - 前端 JS：實作 `clearTime` 函式，呼叫 API 將時間重設為空，並即時更新介面上的進度。

### 整筆任務刪除功能 (2026-03-25)
- **需求**：支援一鍵刪除特定航次的所有進出港任務紀錄（主任務刪除）。
- **實作**：
  - 新增後端 API `POST /{voyage_id}/purge`，使用 SQLAlchemy 的 `delete()` 批次移除該航次所有 Log。
  - 在 `list.html` 的操作列新增「🗑 刪除」按鈕。
  - 在 `detail.html` 的標題工作列新增「🗑 刪除整筆任務」按鈕。
  - 加入 `confirm()` 彈窗確認，防止誤刪。

### 介面強化：DataTables 與流水號 (2026-03-25)
- **需求**：表前面增加編號欄位，並改用 jQuery DataTable。
- **實作**：
  - 在 `base.html` 導入 jQuery 與 DataTables CDN 資源（含繁體中文語系）。
  - 修改 `list.html` 與 `detail.html`：
    - 表格新增 `#` 欄位，使用 Jinja2 `loop.index` 顯示流水號。
    - 使用 `$(table).DataTable()` 初始化，提供搜尋、排序、分頁與單頁筆數切換功能。
  - 提升了大量數據下的瀏覽效率與視覺專業感。

### 排序優化：同類別分組 (2026-03-25)
- **需求**：新增/初始化後，同類別的項目應緊鄰，且具備邏輯順序。
- **實作**：
  - 修改 `routers/voyage_tasks.py`：將初始化與查詢邏輯改為三層排序：
    1. `task_group` (分組名稱)
    2. `display_order` (顯示權重)
    3. `name` (項目名稱)
  - 修改 `detail.html`：調整 DataTable 預設排序為 `[0, 'asc']` (# 欄位)，確保頁面呈現完全依照後端組好的邏輯順序。
  - 改善了資料的一致性，讓使用者在檢核項目時能按「分組區域」連續輸入，減少跳轉。

### 進階互動：時間重置與排序修正 (2026-03-25)
- **需求**：紀錄 Modal 需有重置功能，且 DataTables 的時間排序需正確。
- **實作**：
  - **重置時間**：
    - 在 `recordModal` 新增「重置時間」按鈕。
    - 點擊時呼叫 `clearTime()` API 並使用 DataTable 的 `cell().invalidate()` 確保介面與快取同步更新而不需重整頁面。
  - **排序修正**：
    - 在時間欄位導入 `data-order` 屬性，提供標準 YYYY-MM-DD HH:MM 格式。
    - 對於「尚未執行」項目，`data-order` 設為 `9999-12-31`，使其在升冪排序時正確置於最後。
  - **UI 連動**：修正 `openRecordModal` 邏輯，當已有時間時會自動解析並填入 Input，同時顯示重置按鈕。
